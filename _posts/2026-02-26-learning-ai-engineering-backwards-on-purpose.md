Most people start learning AI Engineering by mastering a programming language first. Only then do they begin building projects and apps.

I decided to do the opposite.

Tutorial hell sounds like, well, absolute hell to me. And I don't want to be stuck there.

I want to build functional projects right away that I can use in my life.

## Why I'm Learning This Way

I'm currently transitioning from a Performance Marketing Analyst to an AI Engineer. With everyday responsibilities already on my plate, I need to find a way to make the most of my learning time.

I came across one of [Louis-François Bouchard's](https://towardsai.net/) posts on my LinkedIn feed last year. It was about learning Python as a practical tool for building AI applications. The project-first approach caught my attention, and I decided to enroll.

![Course dashboard showing Section 1 complete](/assets/img/posts/learning-ai-backwards-section1/course-dashboard.png)
_Four lessons in and already thinking differently about how to learn._

This top-down, project-first approach fits my learning style far better than traditional programming language study. In e-commerce advertising, we create campaigns to test first and then refine based on data. Likewise, I prefer to build something functional first and learn the underlying concepts as needed.

That same 'build first, analyze later' energy is what I'm bringing to AI Engineering.

Section 1 of the course introduces how Large Language Models (LLMs) can help speed up my learning. They can act as a coding assistant, a debugging partner, and an interactive tutor. I plan to lean on all three as I build.

## What I Built in Section 1

I don't want to mindlessly move from one module to another. Every time I learn something new, I want to create something out of it.

With the help of AI tools, I was able to produce some tangible outputs:

- Notes on the introductory videos and CS fundamentals module.
- `python-ai-foundations` and `ai-research-assistant-v1` repositories on GitHub.
- A local Python virtual environment.
- A `.env.example` file for environment variables.
- An initial Project Goal draft in the README.

![Both GitHub repositories side by side](/assets/img/posts/learning-ai-backwards-section1/github-repos.png)
_Two repos, two purposes. Separating learning from building was the first real engineering decision I made._

NotebookLM is my go-to tool for creating notes from different sources. You can upload YouTube videos, documents, or website URLs and it will synthesize them into something digestible. I tested the free tier and was satisfied with what it produced.

For brainstorming my Capstone project, I turned to [ChatGPT Plus](https://chat.openai.com/). It also walked me through how to set up my GitHub repositories and local Python environment.

One thing worth mentioning though is that I intentionally didn't install most packages yet. I briefly experimented with pandas out of curiosity. But my rule is simple: I won't install anything I'm not actively using. Every dependency should have a purpose.

## What This Taught Me

### 1. Creating Something Made It Stick

It's easy to consume courses, going from one tutorial to another. But that progress bar doesn't mean anything if you forget everything right away.

Using [NotebookLM](https://notebooklm.google.com/), I produced summaries, slide decks and mindmaps from the course material. I also made some quizzes to test whether I understood the concepts from what I watched.

![NotebookLM workspace showing sources and study guides](/assets/img/posts/learning-ai-backwards-section1/notebooklm-workspace.png)
_This is what active learning looks like. The course videos are the input and the synthesis, mind maps, and study guides are the output._

Even as I write this article, I am constantly referring back to the notes I have created and reviewing the concepts.

ChatGPT helped me think through the Project Objectives. And the process of answering its questions forced me to think clearly about things I hadn't fully defined yet:

- What exactly is this AI Research Assistant?
- What problem does it solve?
- What is not included in version one?
- What does success look like?

Without that clarity, I might have wasted my time coding and building an app I don't need. And probably end up with a messy, unfinished project that doesn't work.

Creating something out of what I learned made it stick in a way that passive watching never did.

### 2. I Started Before I Was Ready

Before starting this course, Windows 10 stopped receiving security updates on my laptop. Unfortunately, my hardware couldn't support Windows 11. So I switched to Linux Mint.

I had never used Linux in my life. But I didn't let that stop me.

I asked ChatGPT to give me step-by-step instructions on how to install VS Code and set up a local Python environment on Linux Mint. I followed the instructions, encountered some things I didn't understand, asked clarifying questions, and eventually made it work.

![VS Code running on Linux Mint with terminal open](/assets/img/posts/learning-ai-backwards-section1/vscode-linux-mint.png)
_Setting up a Python environment on an OS I had never used before. The terminal at the bottom is proof it actually worked._

That experience set the tone for how I'm approaching this entire learning journey.

You will almost never feel fully ready before starting something new. There will always be an unfamiliar concept, a tool you've never used, a technical hurdle you don't know yet how to clear. Waiting until you feel ready is just a polished version of not starting.

What actually moves you forward is taking the first step anyway and trusting that you'll figure out the rest as you go.

ChatGPT didn't make it effortless for me, but it lowered the barrier enough that "I don't know how" was no longer an excuse.

If I had waited until I felt ready, I'd still be waiting.

### 3. The Engineer in Me Was Always There

Louis consistently encourages his students to build in public. Following his advice, I used ChatGPT Plus to create public repositories in GitHub.

![Louis' email encouraging students to build in public](/assets/img/posts/learning-ai-backwards-section1/louis-email.png)
_The nudge that pushed me to build publicly instead of learning quietly. Accountability changes everything._

These repos don't really have anything flashy yet. They only contain README files, skeletal folders and some initial code. But they're there, they're public, and I know they'll grow as I progress through the course.

I studied Chemical Engineering in college. I wasn't able to practice it professionally. And life took me into marketing instead.

But somewhere in those undergrad years, a certain way of thinking got wired in without me fully realizing it:

- Defining project objectives before building an app.
- Structuring folders and files cleanly before adding any code.
- Keeping my learning environment separate from my project environment.

I thought I was just being careful. Looking back, I realized I was thinking like an engineer.

I could have easily asked ChatGPT to just write the code for me and skipped all the setup. But something felt wrong about that.

Chemical engineering taught me that systems have to be designed before they're built. You don't improvise a chemical process. You define the inputs, the environment, the constraints, then you build. AI engineering isn't that different.

Honestly, none of these felt exciting at the moment: setting up a virtual environment, creating a `.env.example` file, keeping repositories separate. This isn't the kind of work that gets likes online.

But I knew skipping these would cause problems later. Things get complicated fast without a solid foundation. I learned that from engineering and I wasn't about to forget it just because I was starting over as a beginner.

Marketing taught me to optimize. Engineering taught me to design. I spent years doing the first one, but the second one never really left.

Turns out, I was always an engineer. I just took a long detour through marketing to get back here.
